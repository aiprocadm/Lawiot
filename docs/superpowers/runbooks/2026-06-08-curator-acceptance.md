# Runbook куратора: приёмка сид-корпуса (Шаг 10)

Операционный сценарий «от пустой БД до опубликованного акта». Курирование и **публикация —
действия человека-куратора** через Django admin; автоприём НИКОГДА не публикует сам
(спецификация §3/§13). Подробности источника — в
`docs/superpowers/notes/2026-06-08-real-fixtures-characterization.md`.

## Развёртывание
1. `docker compose up --wait` — `web` + `postgres` + `qcluster` подняты и healthy.
   (Для локальной разработки достаточно `.venv` + контейнер `lawiot-db`; `DATABASE_URL` в `.env`.)
2. `python manage.py migrate`
3. `python manage.py seed_corpus` — заводит метаданные актов корпуса (без текста/редакций).
   Идемпотентно: повторный запуск только обновляет.
4. `python manage.py createsuperuser` (если ещё нет), войти в `/admin/`.

## Приём ТК РФ
5. Запустить приём (создаёт ЧЕРНОВИК редакции, не публикует):
   ```
   python manage.py ingest_url --slug tk-rf \
     --url "http://pravo.gov.ru/proxy/ips/?doc_itself=&nd=102074279&page=1&rdk=0"
   ```
   Ожидаемо: `Job #N: success`, ~492 КБ скачано, разобрано ~501 узел
   (15 разделов + 62 главы + 424 статьи), создан черновик, ~15 предложенных связей.
   - При сбое парсера/источника — карантин (FAILED-job + сохранённый RawSource) и/или
     форма ручного импорта в admin (вставить текст/файл). Лог приёма — в `IngestionJob`.

## Ревью и публикация (человек)
6. Открыть черновик редакции в admin: проверить оглавление (разделы→главы→статьи),
   число статей (≈424), заголовок, реквизиты, предложенные связи (`suggested`).
   - Подсказка реквизитов из парсера (`detected_number`) ненадёжна (наивный «первый
     N-ФЗ») — `official_number` выставляет куратор (для ТК РФ: `197-ФЗ`).
7. При кривом разборе — admin-действие «переразобрать из RawSource» (reparse), либо
   правка статей вручную. (reparse защищён от обнуления: 0 статей при наличии прежних → отказ.)
8. Если редакция не первая — сверить diff «черновик ↔ текущая».
9. **Опубликовать** редакцию в admin (становится `is_current`, обновляется поисковый
   индекс). Это сознательно ручной шаг — гарантия юридической корректности.
10. Подтвердить корректные `suggested`-связи (→ `confirmed`), отклонить ложные.
    Читателю видны только `confirmed`; куратору — и `suggested`.

## Приёмочный чек-лист (после публикации, end-to-end)
- [ ] `/search/?q=увольнение` находит ТК РФ, сниппет с подсветкой, deep-link в статью.
- [ ] Страница акта: реквизиты, оглавление с якорями (`#st-81`), текст, панели связей.
- [ ] Внутрикорпусная ссылка кликабельна и ведёт к нужной статье.
- [ ] Утратившая силу редакция доступна в истории; текущая помечена.

## Включение авто-приёма (после успешной приёмки)
11. Выставить `auto_ingest=True` у ТК РФ (в `documents/seed/labor_law.py` И в БД), затем
    `python manage.py ensure_sweep_schedule` — ежедневный sweep (django-q2, брокер в БД)
    создаёт ЧЕРНОВИКИ на изменения источника (по хэшу контента). Публикация — снова вручную.
12. Расширять корпус по одному акту: добавить запись в `SEED_ACTS` (реквизиты/URL — с
    официального источника, НЕ из КонсультантПлюс/Гарант), `seed_corpus`, повторить шаги 5–10.

## Известные ограничения / отложено
- **PDF (Фаза 3):** не реализовано — ТК РФ доступен как HTML. Реализуется, когда в корпус
  войдёт акт, который источник отдаёт только PDF (захватить `capture_fixture`, заполнить
  `test_content_to_text_extracts_pdf`). См. характеризацию.
- **Лог приёма** пишет «Разобрано статей: N», где N — все узлы структуры (статьи+главы+
  разделы), а не только статьи. Косметика; на корректность данных не влияет.
- **`detected_number`** — подсказка, не автозаполнение (см. шаг 6).
