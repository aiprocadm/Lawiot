# Дизайн: разбор подзаконных актов (постановления и приказы)

**Дата:** 2026-06-17
**Статус:** утверждён пользователем, готов к плану
**Ветка:** `feature/lawiot-subordinate-acts-parsing`

## Контекст и цель

Стратегическое решение пользователя: сделать Lawiot **лучшим справочником по
трудовому праву** в узкой нише (а не догонять Гарант/Консультант/Контур по
объёму всего законодательства). Первый шаг к этому — **полнота корпуса трудового
права**: без подзаконных актов (постановления Правительства, приказы Минтруда)
«трудовое право» неполное, и любой будущий AI-ассистент, обученный на неполном
корпусе, будет промахиваться.

Полнота корпуса распадается на три независимых куска:

1. **Разбор подзаконных актов** — научить парсер структуре «пункт/приложение». ← *эта спека*
2. Источники в масштабе — где брать и как заводить десятки актов (отдельная задача).
3. Связи «подзаконка ↔ ТК» — навигация (отдельная задача, частично уже есть).

Начинаем с куска №1, потому что это чистая кодовая работа, она ограничена и
проверяема, и без неё остальные два бессмысленны: если скормить системе
постановление сейчас, кодексовый парсер (`Статья N`) его искалечит.

### Текущее состояние кода (факт)

- `documents.models.Document.DocType` **уже** содержит `CODE`, `FEDERAL_LAW`,
  `DECREE` (Постановление), `ORDER` (Приказ), `OTHER`. Новое поле типа не нужно.
- `documents.models.Article.Kind` содержит только `SECTION`, `CHAPTER`,
  `ARTICLE`. Вида «пункт»/«приложение» нет.
- `ingestion/parsing.py` жёстко настроен на кодекс:
  `ARTICLE_RE = ^Статья\s+(\d+...)`, плюс `SECTION_RE`/`CHAPTER_RE`
  (с `re.IGNORECASE`, римские/арабские номера). Пункты не распознаёт.
- Дерево структуры строится через self-FK `Article.parent` — поддерживает
  произвольную глубину (это переиспользуем).
- Якоря генерируются в `Article.save()` через `_ANCHOR_PREFIX`
  (`section→razdel`, `chapter→glava`, `article→st`).

## Охват v1

Поддерживаем разбор **постановлений Правительства и приказов министерств** — это
нормативные акты с чёткой структурой «пункты/приложения», покрываемые одним новым
парсером. **Письма** Роструда/Минтруда сознательно вне v1 (часто бесструктурная
проза — отдельная грязная задача).

## Выбранный подход (A): отдельный разборщик, выбор по типу документа

Кодексовый разбор не трогаем (он проверен на ТК РФ — 538 статей). Для типов
`DECREE`/`ORDER` добавляем **второй** разборщик. Какой применить — система решает
**по `doc_type`** документа (он известен на момент приёма, его ставит куратор/сид).

Отвергнутые альтернативы:
- **B (один умный парсер на всё):** единая функция ищет все маркеры сразу. Гибче
  для смешанных документов, но это одна большая запутанная функция, риск сломать
  работающий разбор ТК и много ложных срабатываний «1.» в прозе. Отклонено.
- **C (минимум, без дробления на пункты):** хранить сплошным текстом. Крошечно и
  безопасно, но нельзя сослаться на «пункт 5 Постановления», нет якорей и точного
  поиска по пунктам — убивает саму идею справочной системы. Отклонено.

## Часть 1 — модель данных (минимум, низкий риск)

Добавляем два значения в `Article.Kind`:

- `POINT = "point", "Пункт"`
- `APPENDIX = "appendix", "Приложение"`

И две записи в `Article._ANCHOR_PREFIX`:

- `"point": "p"` → якорь пункта вида `p-1-1`
- `"appendix": "pril"` → якорь приложения вида `pril-1`

Миграция: расширение `choices` у поля `kind` Django оформляет как тривиальный
`AlterField` (данные не меняются). Других изменений модели нет.

## Часть 2 — новый разборщик (`parse_points`)

Вход: тот же нормализованный текст, что и у кодексового пути (после
`html_to_text`). Выход: тот же список `ParsedArticle`
(`kind`/`number`/`title`/`text`/`order`/`parent_order`), который потребляет
`create_draft_from_parsed` в `ingestion/services.py` (привязка к родителю — по
индексу `parent_order`, как сейчас у разделов/глав). То есть встраивается в
существующий конвейер без изменения «приёмника».

Разборщик идёт по строкам и распознаёт:

1. **Приложение** — строки вида `Приложение N`, `Приложение к постановлению…`,
   `УТВЕРЖДЕНО/УТВЕРЖДЕНЫ/УТВЕРЖДЁН (постановлением…)`. Становится контейнером
   (`kind=APPENDIX`); последующие узлы вкладываются под него.
2. **Раздел / Глава** — **переиспользуем** существующие `SECTION_RE`/`CHAPTER_RE`
   (Положения внутри приложений часто имеют разделы и главы). Вкладываются под
   текущее приложение, иначе на верхний уровень.
3. **Пункт** — номер с точками в начале строки:
   `^(\d+(?:\.\d+)*)\.\s+(.*)$` → `1.`, `1.1.`, `12.3.4.`. Вложенность строится
   **по дроблёному номеру**: «1.1» — потомок «1», «1.1.1» — потомок «1.1». Если
   родителя по префиксу не нашли — привязка к ближайшему контейнеру
   (приложение/раздел/глава) или верхнему уровню.
4. **Преамбула** (например `Правительство Российской Федерации постановляет:`) —
   в отдельный узел не дробится; остаётся в полном тексте акта (`Redaction.full_text`),
   значит ищется, но не имеет своего якоря. Приемлемо для v1.

**Защита от ложных пунктов.** Пунктом считается только дроблёный номер **в начале
строки** (нормализованный конвейер уже разбивает структурные единицы по строкам,
как для `Статья N`). Точную калибровку даёт живой пример (см. Часть 4).

**Встраивание выбора по типу.** Развилка — в `parse_text(text)` (текстовый вход,
который уже зовут `parse_document` → `parse_text` → `parse_structure`):

```
if doc_type in (DocType.DECREE, DocType.ORDER):
    articles = parse_points(text)
else:
    articles = parse_structure(text)   # существующий кодексовый путь
```

`doc_type` пробрасывается из принимаемого `Document` через
`parse_document(content, content_type, doc_type=...)` → `parse_text(text, doc_type=...)`
(небольшая проводка параметра — сейчас эти функции типа не знают; по умолчанию
`None` → кодексовый путь, обратная совместимость).

**Плоский акт — не ошибка.** Если пунктов не нашли (короткий приказ), документ
сохраняется целым текстом (`full_text`), не «ломается». Две существующие защиты в
`ingestion/services.py` считают только `kind=ARTICLE` и потому неверны для
подзаконки — их **смягчаем**:

- `_article_count(redaction)` сейчас = `articles.filter(kind=Article.Kind.ARTICLE).count()`.
  Для подзаконки должен учитывать и `POINT` (содержательные единицы), иначе
  постановление из одних пунктов считается «пустым».
- `_is_safe_to_publish` (гейт авто-публикации, `AUTOPUBLISH_MIN_RATIO`) опирается
  на `_article_count` → автоматически чинится правкой выше.
- Reparse-guard «переразбор дал 0 статей там, где они были» (#1281,
  `ArticleCountDroppedToZero`/аналог) — проверить, что он использует
  обновлённый счётчик и не отвергает законно-короткую подзаконку.

## Часть 3 — показ на странице

Страница акта (`templates/documents/document_detail.html`) уже рисует структуру
деревом (рекурсивно по `parent`/`children`). Доработка:

- **Пункт** — строка «`N.M.` текст» (у пунктов обычно нет заголовка, только номер
  + текст).
- **Приложение** — заголовок-контейнер «Приложение N: …», внутри его пункты/разделы.
- Якоря уже генерируются в `Article.save()` (Часть 1) — прямые ссылки на пункт
  работают без отдельной доработки шаблона, кроме вывода `id=anchor`.

Паспорт-блок карточки (счётчики структуры через `Counter(a.kind ...)`,
фича #36) сам начнёт считать пункты/приложения; русские подписи берутся из
`get_kind_display()`.

## Часть 4 — фикстуры и тесты (де-риск)

Главный приём против «настроили на придуманную структуру» — разбор по живому
образцу, как было с ТК РФ:

1. **Живой пример.** Существующей командой `capture_fixture` сохраняем один
   настоящий постановление и один настоящий приказ с pravo.gov.ru ИПС
   (`doc_itself&nd=...&print=1`) в `ingestion/fixtures_raw/`.
2. **Характеризация.** Заметка в `docs/superpowers/notes/` описывает их реальную
   структуру (сколько пунктов, есть ли приложения, особенности нумерации).
3. **Приёмочный тест** (по образцу `ingestion/tests/test_real_fixtures.py`):
   на живом примере разобралось ≥N пунктов, ≥K приложений, 0 «сирот»
   (100% привязано к родителю), не опубликовано.
4. **Юнит-тесты** на синтетических минимальных примерах по методике TDD
   (RED→GREEN): пункт, под-пункт, приложение, плоский акт, ложный «1.» в прозе.

### Зависимость (флагируется заранее)

Чтобы взять живой пример, скорее всего понадобится **`nd=`-адрес от
пользователя** на одно постановление и один приказ — headless-поиск по
pravo.gov.ru у Claude не работает (известное ограничение: интеллисёрч —
stateful JS-фреймсет). Всплывёт на этапе плана/реализации, не блокирует дизайн.

## Сознательно вне scope (YAGNI) v1

- **Письма** Роструда/Минтруда (бесструктурная проза) — позже.
- **Дробление под-подпунктов** «а) б)», «1) 2)» — остаются внутри текста пункта.
- **Таблицы и бланки-формы** в приложениях — хранятся как текст, не как структура.
- **Авто-определение реквизитов** (орган, номер, даты-идентичность) — ставит
  куратор/сид (надёжнее для юр. точности); парсер может лишь подсказывать. Дату
  редакции парсер по-прежнему авто-извлекает (`detect_redaction_date`).
- **Источники в масштабе** (кусок №2) и **связи подзаконка↔ТК** (кусок №3) —
  отдельные задачи.

## Затрагиваемые компоненты

- `documents/models.py` — `Article.Kind` +2 значения, `_ANCHOR_PREFIX` +2 записи; миграция.
- `ingestion/parsing.py` — новая `parse_points`; проводка `doc_type` в `parse_document`.
- `ingestion/services.py` — смягчение «ноль статей = брак» для подзаконки.
- `templates/documents/document_detail.html` — отрисовка пункта/приложения.
- `ingestion/fixtures_raw/` + `docs/superpowers/notes/` — живые фикстуры и характеризация.
- Тесты: `ingestion/tests/` (юнит + приёмочный на живой фикстуре).

## Риски

- **Реальная структура неизвестна до фикстуры** → митигируется методом
  «фикстура сначала»; дизайн парсера может потребовать правки после живого HTML
  (приемлемо, флагировано).
- **Зависимость от `nd=`-адреса** пользователя для фикстур.
- **Защита «ноль статей»** требует аккуратного смягчения, чтобы не пропускать
  настоящий мусор (страницы-ошибки источника) — тест на это.

## Критерии готовности

- Постановление и приказ с живых фикстур разбираются в дерево
  пунктов/приложений, 0 «сирот», деплинк на пункт работает.
- Кодексовый разбор ТК РФ не регрессировал (существующие приёмочные тесты зелёные).
- Плоский короткий приказ сохраняется без «брака».
- Весь набор тестов зелёный, `ruff check` чист.
