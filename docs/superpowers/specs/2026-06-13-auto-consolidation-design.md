# Дизайн: Автоматическая консолидация (auto-publish свежей редакции)

Дата: 2026-06-13. Раздел спеки: §17 «Полуавтоматическая консолидация».
Статус: одобрен (brainstorming, 2026-06-13).

## 1. Задача

Ежедневный авто-обход pravo.gov.ru (готов в Плане 3c) уже скачивает **сводный**
текст акта с ИПС-эндпоинта `doc_itself&print=1`, обнаруживает изменение по хэшу и
создаёт **черновик** новой редакции. Сейчас на этом конвейер останавливается:
текст публикует куратор вручную.

Цель фичи — закрыть последний шаг: когда обход находит новую сводную редакцию,
система **сама публикует её как текущую**, без человека. Это и есть «консолидация»
в духе §4 (упрощённая консолидация — принимаем готовый сводный текст официального
источника, **не** сливая поправки сами).

Решение принято осознанно (brainstorming 2026-06-13): основатель ранее снял
ограничение §13 «публикует только человек» и авторизовал авто-публикацию.

## 2. Что уже есть (не трогаем по сути)

- `ingestion/scheduling.py` — `iter_targets` (документы с `auto_ingest=True` и
  непустым `source_url`), `sweep_targets`, `run_sweep`; крон django-q2 в 03:00.
- `ingestion/services.py:ingest_target` — конвейер по одной цели: fetch → hash →
  change-detection → parse → `create_draft_from_parsed` → `extract_links`.
- `Redaction.publish()` — атомарно снимает `is_current` с прежней, ставит
  `review_status=published`, `is_current=True`, `published_at=now`, переиндексирует
  поиск.
- `/changes/` — лента опубликованных редакций по `published_at`.
- Reader-diff `GET /doc/<slug>/diff/<from_pk>/` — сравнение редакций для читателя.
- Очередь ревью в админке — показывает черновики (`review_status=draft`).

Авто-публикация просто запускает всё это готовое.

## 3. Известные дефекты, которые фича обязана починить

1. **Неверная дата.** `ingest_target` зовёт `create_draft_from_parsed` без
   `redaction_date`, поэтому дата по умолчанию = `timezone.now().date()` (день
   приёма), а не реальная «действует с» сводной редакции. До auto-publish это была
   косметика (куратор правил дату руками); при auto-publish это пойдёт читателям —
   значит, дату надо извлекать.
2. **Дубли черновиков.** `create_draft_from_parsed` идемпотентен по
   `(document, redaction_date)`. С датой = «сегодня» каждый прогон на изменившемся
   содержимом плодит новую строку на каждый день. С реальной датой повтор той же
   редакции даёт SKIPPED, а настоящая новая редакция — ровно одну новую строку.

## 4. Решение

Подход: **публикация внутри конвейера** (Approach 1 из brainstorming), пер-актовый
флаг. Четыре точечных изменения, новых приложений нет.

### 4.1. Извлечение даты редакции — `ingestion/parsing.py`

Новая чистая функция:

```
def detect_redaction_date(text: str) -> date | None
```

**Стратегия (подтверждена на реальной фикстуре `tk_rf_real.html`, 2026-06-13).**
ИПС `doc_itself` НЕ даёт единого маркера «действует с» для всего акта; вместо этого
каждая инкорпорированная поправка размечена цитатой закона вида
«(В редакции федерального закона от DD.MM.YYYY № NNN-ФЗ)» (в фикстуре ТК РФ — 138
уникальных таких дат). Берём **максимум** этих дат как дату редакции: это
дата последней инкорпорированной поправки — корректный монотонный прокси
(добавилась поправка → максимум сдвинулся вперёд → новая редакция).

Алгоритм:
```
RX = r"от (\d{2})\.(\d{2})\.(\d{4})\s*(?:№|N)\s*\d+-Ф[КЗ]+"   # IGNORECASE
detect_redaction_date(text) = max( date(y,m,d) for d,m,y in RX.findall(text) )  или None
```
Привязка к шаблону «… № NNN-ФЗ» исключает посторонние даты из тела статей (там
1641 «голая» дата, но только 138 — реальные цитаты законов). На фикстуре ТК РФ
максимум = 2025-12-29. Разбор идёт по уже декодированному (windows-1251) тексту,
как в `parse_document`. Нет ни одной цитаты-закона — возвращаем `None` (не угадываем;
для таких актов авто-публикация не сработает, черновик уйдёт в очередь ревью).

Замечание: это дата ПОДПИСАНИЯ последнего изменяющего закона, а не официальная
«редакция действует с» (последняя может быть позже — когда нормы вступают в силу).
Для датировки истории и идемпотентности этого достаточно: важна монотонность и
стабильность, а не юридическая точность «вступления в силу».

Подключается в `parse_document`: `ParsedDocument` получает поле
`detected_redaction_date: date | None` (рядом с существующими `detected_number`/
`detected_date`-хинтами). Это **не** перетирает ничего автоматически — значение
использует только шаг авто-публикации.

### 4.2. Флаг `auto_publish` — `documents/models.py`

```
auto_publish = models.BooleanField(default=False,
    help_text="Авто-публиковать свежую редакцию из source_url без куратора.")
```

Отдельный от `auto_ingest` (лестница доверия: сначала включаем приём, убеждаемся в
парсере на акте, потом — авто-публикацию). Миграция. В `DocumentAdmin`:
`list_display`/`list_editable`/`list_filter` дополняются `auto_publish`.

### 4.3. Защитный гейт — `ingestion/services.py`

```
AUTOPUBLISH_MIN_RATIO = 0.8  # доля статей от текущей редакции

def _is_safe_to_publish(new_redaction, current_redaction) -> bool
```

Возвращает `False` (→ не публиковать), если:
- у новой редакции 0 статей **и** пустой/тривиальный `full_text`; либо
- текущая редакция существует и число статей новой `< AUTOPUBLISH_MIN_RATIO *`
  числа статей текущей (защита от обрезанного/ошибочного ответа источника — напр.,
  ИПС вернул HTML-ошибку или усечённый текст).

Если текущей редакции нет (первая публикация акта) — гейт пропускает при ≥1 статье.
Порог `AUTOPUBLISH_MIN_RATIO` — модульная константа, легко правится.

### 4.4. Шаг авто-публикации — `ingest_target`

После `create_draft_from_parsed(... redaction_date=parsed.detected_redaction_date)`
и `extract_links_for_redaction`:

```
if target.document.auto_publish
   and parsed.detected_redaction_date is not None
   and _is_safe_to_publish(redaction, current):
       redaction.publish()
       extract_links_for_redaction(redaction)   # после публикации: самоссылки резолвятся
       log "Авто-опубликована редакция #..."
else:
       log "Авто-публикация пропущена: <причина>"  # черновик остаётся в очереди ревью
```

`current` — текущая опубликованная редакция документа (если есть), берётся до
создания черновика.

## 5. Поток данных (ежедневный sweep)

```
run_sweep → sweep_targets → для каждого auto_ingest-документа → ingest_target:
  fetch → hash → изменилось? → store raw → parse (текст + структура + ДАТА)
        → create_draft(redaction_date = detected_redaction_date) → extract_links
        → ЕСЛИ auto_publish И дата есть И гейт пройден:
              publish() → re-extract_links → job=SUCCESS («авто-опубликовано»)
          ИНАЧЕ:
              черновик остаётся → job=SUCCESS («авто-публикация пропущена: причина»)
                                → черновик виден в очереди ревью
```

При авто-публикации бесплатно срабатывает готовое: запись в `/changes/`
(`published_at`), reader-diff против прошлой опубликованной редакции,
переиндексация поиска.

## 6. Обработка ошибок

| Ситуация | Поведение |
|---|---|
| Сеть/парсинг упали | FAILED-job, сырьё в карантине (как сейчас) |
| Гейт не пройден / дата не извлеклась | Черновик НЕ публикуется, живой текст не трогаем, job=SUCCESS с предупреждением, черновик — в очередь ревью куратору |
| Та же дата уже опубликована (`PublishedRedactionExists`) | Ловим в `ingest_target`, job=**SKIPPED** «редакция от DATE уже опубликована» (не ошибка) |

Защитный гейт — не «человек в контуре по умолчанию», а аварийный тормоз против
публикации мусора. Нормальный путь — без человека.

## 7. Тестирование

Гонять весь репозиторий (`ruff check .` + `pytest` без путей — см. lint-scope).

- `detect_redaction_date`: верная дата на реальной фикстуре ТК РФ; `None`, когда
  маркера нет.
- `_is_safe_to_publish`: 0 статей / меньше порога / равно / больше / нет текущей.
- `ingest_target` c `auto_publish=True`: публикует на безопасном входе; оставляет
  черновик на небезопасном/без даты; SKIPPED при совпадении даты.
- Интеграция: после авто-публикации `/changes/` содержит запись, reader-diff
  доступен против прошлой опубликованной редакции, поиск находит новый текст.
- Идемпотентность: два прогона на одном содержимом → одна редакция, второй SKIPPED.

## 8. Выкатка

- `auto_publish` по умолчанию **выключен везде**.
- Включаем для `tk-rf` в `documents/seed/labor_law.py` **после** dry-run,
  подтверждающего корректное извлечение даты на живом источнике.
- `sout-426-fz` остаётся выключенным (у него и `auto_ingest=False`) до приёмки его
  парсера.
- Откат — снять флаг `auto_publish` (данные не мигрируют необратимо: published-
  редакции остаются, но новые не создаются авто-путём).

## 9. Вне объёма (YAGNI)

- Парсинг языка поправок («статью N дополнить…») и авто-слияние в базовый текст —
  это §3 «автоконсолидация», осознанно отложено.
- Уведомления куратору сверх существующей очереди ревью и `/changes/`.
- Расширение корпуса на новые акты (заблокировано: нужны `nd=`-URL от пользователя).
