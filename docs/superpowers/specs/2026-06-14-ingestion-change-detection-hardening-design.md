# B1 — Закалка авто-обновления: детект новой редакции по нормализованному тексту

**Дата:** 2026-06-14
**Статус:** дизайн (ожидает ревью пользователя)
**Трек:** B (обновление корпуса при новой публикации на pravo.gov.ru)
**Предшествует:** спайк B0 (семантика источника) — выполнен, см. ниже

---

## 1. Проблема и мотивация

Lawiot должен оставаться **актуальным сам** — подтягивать свежую сводную редакцию
акта, когда на pravo.gov.ru публикуется поправка. «Водопровод» для этого уже есть
(`ingest_target` → дедуп по дате → гейт безопасности → авто-публикация; ежедневный
`sweep_targets`). Но триггер «содержимое изменилось» построен **неверно**.

### Свидетельство (спайк B0, 2026-06-14)

Сравнение сохранённой фикстуры `tk_rf_real.html` с **живым** эндпоинтом
`http://pravo.gov.ru/proxy/ips/?doc_itself=&nd=102074279&print=1`:

| Метрика | Фикстура | Живой эндпоинт |
|---|---|---|
| SHA-256 **сырых байт** | `a8ded1…` | `b18e3d…` (**отличается**) |
| Размер | 1 460 048 Б | 1 459 870 Б |
| SHA-256 **нормализованного текста** (`html_to_text`) | `53c204…32d1` | `53c204…32d1` (**идентичен**) |
| Дата редакции (`detect_redaction_date`) | 2025-12-29 | 2025-12-29 |
| Структура | 15/70/538 | 15/70/538 |

**Выводы:**

1. **ИПС отдаёт ЖИВУЮ сводную редакцию** по фиксированному `nd=` — текст уже
   инкорпорировал поправки до конца 2025 г. Когда выйдет новая поправка, сводный
   текст по тому же URL обновится. → **Пассивный `sweep` — верный механизм.**
   Активный мониторинг ленты опубликования (трек B2) для поддержания актуальности
   **не требуется** (B2 нужен только для обнаружения совершенно новых актов — позже).

2. **Дефект:** `content_changed` хэширует **сырые байты**
   (`compute_hash(result.content)` в `ingestion/services.py`). HTML-обёртка ИПС
   содержит несущественный «дребезг» (~178 Б: токен/таймстемп/счётчик в разметке),
   меняющийся между запросами **без всякой поправки**. Поэтому каждый ежедневный
   обход видит «изменилось» (ложно), перекачивает и переразбирает 1.4 МБ, и от
   создания ложной редакции спасает только дедуп по дате (`PublishedRedactionExists`
   → SKIPPED). Сигнал «новая редакция» сейчас — шум.

## 2. Цель

Сделать сигнал «новая редакция» **достоверным**: детект изменения — по хэшу
нормализованного текста, не по сырым байтам. Тогда новая редакция → текст меняется
→ настоящий сигнал; дребезг разметки → текст тот же → корректный SKIP без лишней
ежедневной работы. Плюс — сквозная проверка всей цепочки на реальной 2-й редакции.

## 3. Объём

### В объёме

- `RawSource.text_hash` — хэш нормализованного текста (новое поле, миграция).
- Расщепление `parse_document` на `html_to_text` + `parse_text` (нормализация один раз).
- Хелпер `compute_text_hash(content, content_type)`; `store_raw_source` проставляет
  оба хэша; `ingest_target` решает skip/proceed по `text_hash`.
- Юнит-тест детекта (сценарий спайка) + сквозной django_db-тест на синтетической
  «2-й редакции».

### Вне объёма (осознанно)

- **B2** — активный мониторинг `publication.pravo.gov.ru` (не нужен: источник живой).
- **A-parse-1** — под-статейная гранулярность (части/пункты) — следующий спек.
- Подавление junk-черновиков от страниц-ошибок источника — возможный будущий хвост.

## 4. Дизайн

### 4.1. Модель

```python
class RawSource(models.Model):
    ...
    content_hash = models.CharField(max_length=64, db_index=True)  # сырьё (форензика)
    text_hash = models.CharField(max_length=64, blank=True, db_index=True)  # НОВОЕ
```

Миграция `ingestion/0002_rawsource_text_hash` — **только схема**. Backfill не делаем:
старые строки получают `text_hash=""`, что ≠ любому реальному хэшу → первый
пост-деплойный обход по каждой цели разово засчитает «изменилось», переразберёт,
создаст новый `RawSource` с заполненным `text_hash` → дальше стабильно.

### 4.2. Парсинг (нормализация один раз)

```python
def parse_text(text: str) -> ParsedDocument: ...      # структура+заголовок+реквизиты
def parse_document(content, content_type="text/html") -> ParsedDocument:
    return parse_text(html_to_text(content, content_type))   # тонкая обёртка, back-compat
```

`html_to_text` уже существует и не меняется. Расщепление позволяет в `ingest_target`
нормализовать содержимое **один раз** — и для хэша, и для разбора.

### 4.3. Сервисы

```python
def compute_text_hash(content: bytes, content_type: str) -> str:
    return hashlib.sha256(html_to_text(content, content_type).encode("utf-8")).hexdigest()

def store_raw_source(target_key, content, content_type="", source_url=""):
    return RawSource.objects.create(
        target_key=target_key,
        content=content,
        content_hash=compute_hash(content),                 # сырьё
        text_hash=compute_text_hash(content, content_type), # нормализованный текст
        content_type=content_type, source_url=source_url,
    )

def text_changed(target_key, text_hash) -> bool:
    latest = RawSource.objects.filter(target_key=target_key).order_by("-fetched_at").first()
    return latest is None or latest.text_hash != text_hash
```

В `ingest_target`: после `fetch` вычисляем `text = html_to_text(...)`,
`text_hash = sha256(text)`; решение skip/proceed — по `text_changed(target_key, text_hash)`
(вместо `content_changed`). При proceed — `parse_text(text)` (без повторной
нормализации). Гейт `_is_safe_to_publish`, дедуп по дате, извлечение связей,
авто-публикация — **без изменений**.

### 4.4. Что НЕ меняется

Дедуп-по-дате (`PublishedRedactionExists`), гейт безопасности
(`AUTOPUBLISH_MIN_RATIO=0.8`), извлечение связей, лента изменений, reader-diff,
расписание `sweep_targets`/django-q2 — корректны и остаются как есть.

## 5. Обработка краёв

| Случай | Поведение |
|---|---|
| Дребезг разметки (тот же закон) | Нормализованный текст тот же → `text_changed`=False → SKIPPED. Цель B1. |
| Реальная поправка | Текст меняется + новая `redaction_date` → черновик новой редакции → гейт → (авто-)публикация. |
| Страница-ошибка источника | Текст ≠ реальному → «изменилось» → разбор → гейт ловит 0 статей → **не публикует**, черновик ждёт куратора (не хуже текущего). |
| Ручной импорт / не-HTML | `compute_text_hash` декодирует как текст → строки RawSource единообразны. |
| Микроправка ИПС (опечатка), та же дата | Текст меняется → черновик той же даты; уже опубликованная редакция этой даты → `PublishedRedactionExists` → SKIPPED (данные не трогаются). |

## 6. Тестирование

Новые тесты — в отдельном файле (`ingestion/tests/test_change_detection.py`),
чтобы не задевать конфликт-хотспоты.

**Юнит (без БД):**
- Два HTML, различающиеся только markup-токеном → `compute_text_hash` совпадает.
- HTML с изменённым **текстом статьи** → `compute_text_hash` различается.

**Сквозной (django_db):**
- Синтетическая фикстура-«2-я редакция»: дата поправки `15.01.2026 № 5-ФЗ`
  (`detect_redaction_date` → 2026-01-15), ~то же число статей, одна изменённая.
- Документ с опубликованной R1 (2025-12-29, `auto_ingest`+`auto_publish`).
- `ingest_target` на фикстуре → ожидаем: новый черновик R2 (2026-01-15) → гейт
  пройден → авто-публикация → R1 `is_current=False`, R2 `is_current=True` →
  reader-diff(R1→R2) показывает изменённую статью → лента изменений содержит R2.
- Повторный `ingest_target` тем же содержимым → `text_changed`=False → SKIPPED,
  новый RawSource не создаётся.
- Регресс: существующие тесты `content_changed`/приёма обновляются под `text_hash`.

## 7. Миграция и развёртывание

- 1 схема-миграция `ingestion/0002_rawsource_text_hash` (классификатор миграций
  применяет пользователь на общей dev-БД).
- Первый пост-деплойный `sweep` разово переингестит каждую цель (text_hash="" на
  старых строках) — это ожидаемо и безвредно (дедуп по дате не создаст дублей).

## 8. Риски

- **Двойная нормализация при неаккуратном рефакторинге** — устранена расщеплением
  `parse_text`; следить, чтобы `ingest_target` звал `html_to_text` один раз.
- **Источник сменит формат HTML** так, что `html_to_text` начнёт давать другой
  текст без поправки — даст ложную редакцию; защита — гейт + дедуп по дате (новая
  дата не извлечётся → авто-публикация не сработает, черновик к куратору).
