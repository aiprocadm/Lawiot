# Карточка акта + статусы — дизайн

Дата: 2026-06-15
Статус: одобрено пользователем

## Проблема

Текущая «карточка» документа в `templates/documents/document_detail.html` —
плоский `<p>` с четырьмя полями (тип/номер, орган, статус, дата редакции).
Модель `Document` уже хранит больше реквизитов, которые **не выводятся**:
`sign_date` (дата подписания) и `official_pub_date` (дата опубликования).
Статус показан простым текстом без визуального выделения.

Цель: превратить плоский блок в структурный «паспорт акта» в стиле СПС
(Гарант/Консультант) — с полными реквизитами, цветным бейджем статуса и
кратким обзором структуры, — и заполнить недостающие реквизиты данными.

## Ключевое решение по источнику реквизитов

Наивный детект реквизитов из текста даёт **мусор** (проверено на живой
фикстуре `tk_rf_real.html`): первый токен `NNN-ФЗ` в тексте ТК РФ — это
`97-ФЗ` из первой *изменяющей* поправки «от 24.07.2002 № 97-ФЗ», а не
собственный номер акта `197-ФЗ`; первая дата `ДД.ММ.ГГГГ` = `24.07.2002`
(первая поправка), а не дата подписания `30.12.2001`. Собственный номер
акта у заголовка вообще не стоит.

Вывод: реквизиты-**идентичность** акта (номер, дата подписания, дата
опубликования, орган, тип) по природе **не меняются между редакциями** —
это паспорт акта, и его естественное место — **сид** (`SEED_ACTS`),
заводится один раз. Парсер же надёжно извлекает то, что *меняется* —
`redaction_date` (уже реализовано). Это разделение устраняет весь класс
багов «детект схватил изменяющий закон вместо самого акта».

Поэтому источник реквизитов = **сид** (вариант A из брейншторма). Детект
реквизитов из текста в этой фиче не добавляется.

## Объём работ

### 1. Данные (сид) — без миграций

Поля `Document.sign_date` и `Document.official_pub_date` **уже существуют**
в модели (сейчас `None` у обоих актов). Добавить их с официальными
значениями в `documents/seed/labor_law.py`:

- **ТК РФ (197-ФЗ):** `sign_date = 2001-12-30`,
  `official_pub_date = 2001-12-31` («Российская газета» № 256, 31.12.2001).
- **426-ФЗ (СОУТ):** `sign_date = 2013-12-28`,
  `official_pub_date = 2013-12-30` («Российская газета» № 295, 30.12.2013).

`issuing_body` у обоих уже заполнен. Команда `seed_corpus` использует
`Document.objects.update_or_create(slug=..., defaults=...)`, поэтому
повторный прогон проставит новые даты на уже существующие документы.
Миграций нет — поля присутствуют в схеме.

Значения дат указываются как `datetime.date` в словарях `SEED_ACTS`.

### 2. Вьюха `documents/views.py:document_detail`

Добавить в контекст счётчики структуры одним запросом-агрегатом по уже
загруженной текущей редакции:

```python
from django.db.models import Count
counts = {
    row["kind"]: row["n"]
    for row in redaction.articles.values("kind").annotate(n=Count("id"))
}
# section_count / chapter_count / article_count в контекст
```

Значения по умолчанию `0`, если узлов данного вида нет. Это
не добавляет N+1 — один агрегатный запрос.

### 3. Шаблон `templates/documents/document_detail.html`

Заменить плоский `<p>` (строки заголовка) на структурную карточку
«паспорт акта» через `<dl>`:

- Вид документа (`get_doc_type_display`)
- Номер (`official_number`)
- Принявший орган (`issuing_body`, fallback «—»)
- Дата подписания (`sign_date`, fallback «—»)
- Дата опубликования (`official_pub_date`, fallback «—»)
- **Статус** — цветной бейдж `<span class="status-badge status-{{ status }}">`:
  🟢 действует (`in_force`) / 🔴 утратил силу (`repealed`) /
  ⚪ не вступил (`not_in_force`)
- Действующая редакция (`redaction.redaction_date`)
- Структура: «N разделов · N глав · N статей» (из счётчиков; виды с
  нулём не показывать)

Минимальный CSS для бейджа добавить в `base.html` (проект на Pico CSS):
небольшой инлайн-блок с фоновым цветом по статусу (зелёный/серый/красный),
скруглением и отступами. Блоки «Другие редакции», оглавление, текст и
панель связей (`aside`) — **без изменений**.

### 4. Тесты

`documents/tests/` (новый файл `test_document_card.py`, чтобы не трогать
hotspot `test_views.py`):

- **view:** контекст содержит верные `section_count`/`chapter_count`/
  `article_count` для документа с известной структурой;
- **template:** на отрендеренной странице присутствуют `sign_date`,
  `official_pub_date` и бейдж статуса с правильным классом;
- **fallback:** документ с пустыми `sign_date`/`official_pub_date`
  показывает «—», а не пустоту/ошибку;
- **seed:** после `seed_corpus` у существующего документа проставлены
  `sign_date` и `official_pub_date` из `SEED_ACTS`.

Тесты под Postgres (как и весь django_db-набор).

## Границы (YAGNI)

- Новых полей модели нет (используются существующие `sign_date`/
  `official_pub_date`).
- Поле «вступил в силу» (effective_date) НЕ добавляется (отклонено в
  брейншторме).
- Детект реквизитов из текста НЕ добавляется — источник реквизитов — сид.
- Панель связей, оглавление, список редакций, diff-страница — без
  изменений.

## Риски

- Точность официальных дат публикации: используются даты первой
  официальной публикации в «Российской газете». Если потребуется иной
  официальный источник (СЗ РФ), значение в сиде легко скорректировать —
  это одна строка, без миграции.
