# Дизайн: семантический поиск (AI-срез 4)

Дата: 2026-06-24. Roadmap §3-C, срез 4: «семантический поиск (pgvector + эмбеддинги)
— ловит перефразировки, которых не берёт FTS; гибрид с текущим лексическим поиском».

## Проблема

Лексический FTS (`russian` tsvector + лемматизация + синонимы) находит по
словам/словоформам. Перефразировки без общих лемм он не берёт: «можно ли не выходить
на работу в день рождения» ↔ ст. об отпуске без сохранения зарплаты; «удалёнка из
другого города» ↔ дистанционная работа. Семантические эмбеддинги ловят смысл.

## Стратегия (решения брейншторма)

- **Источник эмбеддингов — локальная модель** `intfloat/multilingual-e5-small`
  (384-dim) через `sentence-transformers`: ноль повторяющейся стоимости, нет
  внешнего ключа/аккаунта, работает офлайн. Корпус мал → CPU-инференс достаточен.
- **Аддитивный supplement, НЕ замена ранжирования.** FTS-результаты сохраняются
  как есть (порядок и тесты не меняются); семантика ДОБАВЛЯЕТ документы, которые
  FTS пропустил целиком. Низкий риск регресса; главный выигрыш — recall на
  перефразировках. (Полноценный RRF-гибрид — возможное будущее улучшение.)
- **Грациозная деградация:** нет модели/пакета или эмбеддинг не посчитался →
  чистый FTS (текущее поведение). Фича никогда не ломает поиск.
- **Эмбеддинги вне request-пути.** Генерируются командой `embed_articles`
  (бэкфилл/после ингеста), не при сохранении в вебе — модель не грузится в
  gunicorn. Запрос на поиск эмбедит ТОЛЬКО короткий текст запроса.

## Не-цели (YAGNI)

- НЕ RRF/переранжирование FTS (v1 — supplement).
- НЕ авто-эмбеддинг при publish (отдельная команда; при желании — в daily-sweep).
- НЕ эмбеддинг редакций целиком (статья — семантическая единица и цель deep-link).
- НЕ дообучение/свой энкодер.

## Архитектура

### 1. Инфраструктура pgvector

- Образ БД `postgres:16-alpine` → **`pgvector/pgvector:pg16`** (та же мажорная
  версия PG, тот же том `lawiot_pgdata` — данные сохраняются; меняется только
  контейнер). Деплой-шаг (документируется в PR): `docker compose up -d db` пересоздаёт
  контейнер с расширением.
- Python-пакет `pgvector` (Django-поле/операторы) — лёгкий, в requirements.
- `sentence-transformers` в requirements (тянет torch). В тестовый/хостовый venv
  НЕ обязателен — тесты мокают бэкенд эмбеддингов.

### 2. Модель и миграция

- `documents/models.py`: `Article.embedding = VectorField(dimensions=384, null=True,
  blank=True, editable=False)` (`pgvector.django.VectorField`). null до бэкфилла.
- Миграция `documents/00NN`:
  - `pgvector.django.VectorExtension()` — `CREATE EXTENSION IF NOT EXISTS vector`.
  - `AddField` embedding.
  - HNSW-индекс `vector_cosine_ops` (`pgvector.django.HnswIndex`) — forward-compat;
    на малом корпусе и точный скан быстр.

### 3. `search/embeddings.py` — бэкенд эмбеддингов

```
EMBED_DIM = 384
def embed_query(text) -> list[float] | None      # с префиксом "query: "
def embed_passages(texts) -> list[list[float]]   # с префиксом "passage: "
def set_backend(fn) / get_backend()              # инъекция для тестов
```

- Ленивый бэкенд: `SentenceTransformer("intfloat/multilingual-e5-small")` грузится
  при первом вызове (lru_cache). e5 требует префиксы `query:`/`passage:`.
- Нет пакета/модели → бэкенд возвращает None (для query) / поднимает понятную
  ошибку (для бэкфилла, где это явная операция). `embed_query` глотает любую
  ошибку → None (деградация на поиске).
- Тесты вызывают `set_backend(fake)` — детерминированные векторы по фразам.

### 4. `search/services.py` — гибрид

После сборки FTS-результатов (без изменений) — `_semantic_supplement`:
1. `vec = embed_query(query_text)`; если None → вернуть FTS как есть.
2. `Article.objects.filter(<current published> + те же doc-фильтры)
   .exclude(embedding=None).order_by(CosineDistance("embedding", vec))[:K]`.
3. Для статей, чей документ ЕЩЁ НЕ в FTS-результатах, собрать `SearchResult`
   (anchor/label статьи, сниппет — простой обрез текста статьи, `rank=None`/0,
   пометка `semantic=True` для шаблона).
4. Добавить их ПОСЛЕ FTS-результатов (FTS-порядок сохранён).

`SearchResult` получает поле `semantic: bool = False` (шаблон может показать
«найдено по смыслу»).

### 5. Команда `embed_articles`

`manage.py embed_articles [--all]`: бэкфилл embedding для статей текущих
опубликованных редакций (по умолчанию только где `embedding IS NULL`; `--all` —
переэмбедить всё). Батчами через `embed_passages`; `bulk_update`. Идемпотентна.

## Обработка ошибок / деградация

- Нет pgvector-расширения в БД → миграция не применится (деплой-шаг обязателен;
  документировано). До миграции код не вызывается на проде.
- Нет модели/пакета → `embed_query`→None → чистый FTS.
- Пустой корпус эмбеддингов (бэкфилл не запускался) → семантический запрос вернёт
  пусто → чистый FTS.
- Запрос с websearch-операторами/в кавычках — семантика всё равно применима
  (эмбедим сырой текст запроса); FTS-ветка обрабатывает операторы как прежде.

## Тестирование

Тесты гоняются против pgvector-БД (`pgvector/pgvector:pg16`); бэкенд эмбеддингов
замокан (`set_backend`), torch не нужен.
- `embeddings`: префиксы query/passage; деградация без бэкенда (None).
- миграция/поле: `VectorField` создаётся, `CosineDistance` сортирует (мок-векторы).
- `search_documents`: семантика добавляет документ, который FTS не нашёл; FTS-порядок
  не меняется; без бэкенда — поведение идентично текущему (регрессионные тесты FTS
  зелёные); doc-фильтры применяются и к семантике.
- `embed_articles`: бэкфилл проставляет векторы; идемпотентность.

## План отката

Аддитивно: новое поле (nullable), модуль, команда, supplement за guard'ом.
Откат — revert PR + (опц.) вернуть образ БД. Без эмбеддингов/без расширения после
отката поиск работает как сейчас (FTS).

## Деплой-шаги (в PR, выполняет user на общей среде)

1. `docker compose up -d db` (пересоздать БД с pgvector; том сохраняется).
2. `manage.py migrate` (расширение + поле + индекс).
3. `pip`/rebuild образа с `sentence-transformers` (или `manage.py embed_articles`
   на хосте с установленной моделью).
4. `manage.py embed_articles` — бэкфилл.
