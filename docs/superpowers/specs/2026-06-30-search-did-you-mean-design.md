# Дизайн: «Вы искали…» — исправление опечаток в поиске

Дата: 2026-06-30. Трек: глубина поиска (roadmap §6.2, первый срез).

## Проблема

Опечатка в поисковом запросе («уволнение» вместо «увольнение») даёт 0 результатов —
тупик для пользователя. FTS с лемматизацией (pymorphy3) не спасает: pymorphy3 разбирает
*известные* словоформы, но не исправляет опечатки (неизвестное слово → пустой/низкий разбор).
Нужен отдельный механизм исправления.

## Решение (кратко)

При **нулевом** результате поиска предложить ближайшее по написанию реальное слово корпуса
как кликабельную ссылку: «Возможно, вы искали: **<исправленный запрос>**». Исправление —
через триграммное сходство Postgres (`pg_trgm`) против словаря словоформ, извлечённого из
текста корпуса.

Решения, зафиксированные на брейнсторме:
- **Триггер:** только при 0 результатов (классический did-you-mean; не раздражаем на хороших
  запросах).
- **Применение:** подсказка-ссылка, не авто-замена (пользователь сам кликает).
- **Источник слов:** реальные словоформы корпуса (не основы из `search_vector`) — чтобы
  подсказывать «увольнение», а не основу «увольнен».
- **Сопровождение словаря:** management-команда (ручной запуск), как соседние `reindex_search`
  и `embed_articles`. Авто-хук на публикацию — отдельный пункт бэклога (roadmap §6.1), в этот
  срез не входит.
- **Движок:** `pg_trgm` + GIN-индекс по триграммам (проект уже работает с расширениями БД —
  pgvector в `documents/migrations/0017`).

## Раскладка по приложениям

Следуем существующей границе: **вся схема БД и data-команды — в `documents/`** (там же
`search_vector`/`embedding` на `Article` и команды `reindex_search`/`embed_articles`);
**логика запроса — в `search/`** (логика-только приложение).

## Компоненты

### 1. Модель `SearchVocab` (`documents/models.py`)

```python
class SearchVocab(models.Model):
    word = models.CharField(max_length=128, unique=True)   # словоформа, lowercase, ё→е
    frequency = models.PositiveIntegerField(default=1)     # частота в корпусе

    class Meta:
        ordering = ["word"]
        indexes = [GinIndex(fields=["word"], name="searchvocab_word_trgm", opclasses=["gin_trgm_ops"])]
```

- `word` нормализован так же, как поисковые токены: lowercase + ё→е.
- `frequency` — тай-брейк при равном сходстве (предпочитаем частотное слово).
- GIN-индекс по `gin_trgm_ops` делает поиск ближайшего слова дешёвым в SQL.

**Миграция** (по образцу `0017` с pgvector):
- `from django.contrib.postgres.operations import TrigramExtension`
- операции: `TrigramExtension()`, затем `CreateModel(SearchVocab)` с `AddIndex` GIN-триграммного.

### 2. Команда `build_search_vocab` (`documents/management/commands/`)

Сборка по образцу `embed_articles` (батчи, идемпотентность, флаги):
- Источник: `Article.text` опубликованных текущих редакций
  (`redaction__is_current=True, redaction__review_status=PUBLISHED`).
- Токенизация: `re.findall(r"[а-яёa-z]+", text.lower())`, затем ё→е.
- Фильтры: длина `>= --min-len` (дефолт **4**), итоговая частота `>= --min-freq` (дефолт **2**) —
  отсекает OCR-мусор и опечатки самого корпуса.
- Запись: идемпотентный полный пересбор — внутри `transaction.atomic()`:
  `SearchVocab.objects.all().delete()` + `bulk_create` (батч 1000).
- Вывод: число обработанных статей и размер словаря (как у `embed_articles`/`reindex_search`).

Флаги: `--min-len` (int, 4), `--min-freq` (int, 2).

### 3. Движок подсказки (`search/suggest.py`)

```python
SIMILARITY_THRESHOLD = 0.3   # дефолтный порог pg_trgm

def suggest_query(query_text: str) -> str | None: ...
```

Алгоритм:
1. Токенизировать запрос той же нормализацией (`[а-яёa-z]+`, lowercase, ё→е). Сохранить порядок.
2. Собрать множество известных слов: `SearchVocab.objects.filter(word__in=tokens)`
   одним запросом → `known: set[str]`.
3. Для каждого токена:
   - если в `known` → оставить как есть;
   - иначе («подозрительный») найти ближайшее:
     `SearchVocab.objects.annotate(sim=TrigramSimilarity("word", token))
        .filter(sim__gte=SIMILARITY_THRESHOLD).order_by("-sim", "-frequency").first()`
     → заменить на `.word`, если найдено; иначе оставить токен без изменений.
4. Если **хотя бы один** токен заменён → вернуть собранную строку (токены через пробел);
   иначе вернуть `None`.

Граница ответственности: возвращает *строку-кандидат* или `None`. Не знает про HTTP/шаблоны.
Не трогает валидные (известные корпусу) слова.

**Деградация:** любые ошибки БД/отсутствие `pg_trgm`/пустой словарь → ловим, логируем,
возвращаем `None` (как `embeddings.embed_query`). Поиск не ломается.

### 4. Интеграция во view (`search/views.py`)

В `search_view`, после получения `results`:
```python
suggestion = None
if cd.get("q") and not results:
    candidate = suggest_query(cd["q"])   # None, если ни один токен не заменён
    if candidate and search_documents(candidate, **filters):
        # Верификация: показываем, только если исправленный запрос даёт результаты.
        suggestion = candidate
```
- Срабатывает **только** на нулевом результате → лишний поиск платится редко.
- Проверка «отличается ли кандидат от запроса» не нужна: `suggest_query` по контракту
  возвращает `None`, когда заменять нечего.
- Верификация гарантирует, что ссылка ведёт не «в пустоту» (все слова известны корпусу, но
  AND-запрос всё равно мог бы дать 0).
- `suggestion` (строка) кладётся в контекст.
- Ссылка сохраняет прочие фильтры: берём `base_qs`, заменяем `q` на `suggestion`.

### 5. Шаблон (`templates/search/_results.html`)

- `page_obj` пуст и есть `suggestion` →
  «Возможно, вы искали: <a href="?{{ suggestion_qs }}">{{ suggestion }}</a>».
- `page_obj` пуст и нет `suggestion` → текущее сообщение «ничего не найдено» (без изменений).
- `suggestion_qs` формируется во view (querystring с заменённым `q`), чтобы шаблон не собирал URL.

## Поток данных

```
[offline] build_search_vocab → SearchVocab (word, frequency) + триграммный GIN-индекс
[runtime] запрос → search_documents → []  ──► suggest_query(q)
              └► TrigramSimilarity по SearchVocab → кандидат
                   └► верификация search_documents(кандидат) > 0
                        └► контекст.suggestion → ссылка в шаблоне
```

## Обработка ошибок

- Пустой словарь / нет `pg_trgm` → `suggest_query` → `None`, лог уровня INFO. Поиск работает.
- Кандидат == исходному запросу (после нормализации) → не показываем.
- Кандидат не даёт результатов при верификации → не показываем.

## Тестирование (`search/tests/test_suggest.py`)

Команда `build_search_vocab`:
- извлекает слова из текста статей, считает частоты;
- нормализует ё→е;
- отсекает слишком короткие (`< min-len`) и редкие (`< min-freq`);
- идемпотентна (повторный запуск → тот же словарь).

`suggest_query`:
- опечатка с близким словом в словаре → исправленная строка;
- все слова известны → `None`;
- нет близкого слова (ниже порога) → `None`;
- многословный запрос: правит только незнакомый токен, известные сохраняет;
- пустой словарь → `None` (без исключений).

View (`search_view`):
- нулевой запрос с опечаткой → в ответе ссылка-подсказка с корректным `q`;
- запрос с результатами → подсказки нет (движок не вызывается);
- ссылка-подсказка несёт остальные активные фильтры;
- кандидат, не дающий результатов → подсказки нет.

## Вне scope (YAGNI)

- Авто-замена / «показаны результаты по Y, искать X» (выбран вариант «ссылка»).
- Подсказки при непустом результате.
- Авто-перестроение словаря на публикацию (отдельный пункт бэклога §6.1).
- Фасеты, фильтры по реквизитам, экспорт результатов (другие срезы §6.2).
- Многоязычность (корпус русскоязычный).
