# Реестр ожидаемых актов (pending-watchlist) — Design

**Дата:** 2026-06-15. **Статус:** утверждено (brainstorming 2026-06-15).
**Контекст:** §3/§17 спеки — расширение корпуса. Появился акт (565-ФЗ «О занятости
населения в РФ»), который мы хотим в корпусе, но **не можем принять сейчас**. Нужен
механизм, который не даст про него забыть и подскажет куратору, что и как сделать,
когда акт станет доступен.

## 1. Задача и почему именно так

При попытке завести 565-ФЗ выяснились три факта (проверено вживую 2026-06-15):

1. **Движок приёма — не блокер.** Ручной импорт уже есть (`import_document`): любой
   акт, чей текст доступен как HTML/TXT, заводится без кода. Блокер — *источник текста*.
2. **565-ФЗ нет в ИПС в консолидированном виде.** Классическая ИПС «Законодательство
   России» (эндпоинт `doc_itself`, на котором работает наш парсер) отдаёт по запросу
   только **отменённого предшественника** (Закон РФ 1032-1, title «Утратил силу»).
   Сам 565-ФЗ доступен лишь на `publication.pravo.gov.ru` как **скан-PDF исходной
   редакции** (1-битные изображения страниц → нужен OCR; и это не консолидированный
   текст). Консолидированные редакции из КонсультантПлюс/Гарант брать нельзя (их ИС, §2).
3. **Авто-поиск по ИПС нереализуем надёжно.** Поиск, «новые поступления» и листинги
   ИПС — JavaScript-приложение (frameset + клиентский JS, в серверном HTML ноль `nd`).
   Автоматизация потребовала бы reverse-engineering недокументированного JSON-API или
   headless-браузера — хрупко и высоко-сопровождаемо, против §2.1 «минимум движущихся
   частей».

**Вывод:** автоматизируем не «поиск» (ИПС его прячет), а «память» — то, что человек
забывает. Реестр ожидаемых актов + всегда видимый список в admin. Находит и заводит
акт по-прежнему человек, через браузер и существующий `ingest_url`.

## 2. Что делаем

### 2.1. Модель `PendingAct` (`documents/models.py`)

Ожидаемый, но ещё не принятый акт.

- `slug` (`SlugField`, unique) — слаг, который акт получит при приёме.
- `title` (`CharField`) — ожидаемое официальное наименование.
- `official_number` (`CharField`) — ожидаемый номер, напр. `565-ФЗ`.
- `doc_type` (`CharField`, те же choices, что у `Document.doc_type`).
- `note` (`TextField`, blank) — почему ждём (напр. «только скан-PDF, ждём появления
  в ИПС в консолидированном виде»).
- `ips_search_url` (`URLField`, blank) — ссылка-подсказка на поиск ИПС (открывается
  в браузере, где JS работает).
- `added_at` (`DateTimeField`, `auto_now_add`).

Свойство `is_resolved` — **вычисляемое, не хранимое**: `True`, когда в корпусе уже
есть `Document` с тем же `official_number` и `doc_type`, у которого есть опубликованная
текущая редакция (`Redaction.is_current=True, review_status=published`). Состояние
выводится из корпуса — никакого ручного флага, который можно забыть переключить.

Миграция (одна). Без изменений других моделей.

### 2.2. Admin (`documents/admin.py`) — это и есть «напоминание»

`PendingActAdmin`:
- `list_display`: `title`, `official_number`, `doc_type`, `is_resolved`, `added_at`.
- `list_filter`: `is_resolved`, `doc_type`. По умолчанию куратор смотрит неразрешённые.
- `ips_search_url` — кликабельная ссылка (в браузер, где JS-поиск ИПС работает).
- Readonly-поле «как завести» (`ingest_hint`): готовый шаблон команды
  `python manage.py ingest_url --slug <slug> --url "http://pravo.gov.ru/proxy/ips/?doc_itself=&nd=<ND>&print=1"`
  с явным местом под `<ND>` и напоминанием «после приёма — ревью и публикация вручную».

Напоминание = постоянная видимость списка в admin (куратор — основной пользователь
admin, §7). Ничего не настраивать (ни SMTP, ни планировщик), невозможно «пропустить».

### 2.3. Seed-интеграция (`documents/seed/labor_law.py`)

- Новый декларативный `PENDING_ACTS` рядом с `SEED_ACTS`. Существующий комментарий-
  список кандидатов заменяется этим списком. Первая запись — **565-ФЗ**:
  `slug=zanyatost-565-fz`, `title=О занятости населения в Российской Федерации`,
  `official_number=565-ФЗ`, `doc_type=federal_law`, заметка про скан-PDF/ИПС,
  `ips_search_url` на поиск ИПС.
- `seed_corpus` после обработки `SEED_ACTS`:
  - идемпотентно создаёт/обновляет `PendingAct` из `PENDING_ACTS` (upsert по `slug`);
  - **удаляет** `PendingAct`, которые уже разрешены (акт заведён и опубликован) —
    список сам себя чистит. Удаление по `is_resolved`, а не по присутствию в
    `SEED_ACTS`: пункт исчезает именно когда акт реально опубликован, а не когда его
    просто перенесли в `SEED_ACTS`.

## 3. Поток данных

```
dev/куратор добавляет акт в PENDING_ACTS (или прямо PendingAct в admin)
  → seed_corpus материализует в PendingAct
  → куратор открывает admin → видит список «Ожидаемые акты» (неразрешённые)
  → по ips_search_url проверяет ИПС в браузере
  → когда акт появился в консолидированном виде: копирует nd,
     запускает ingest_url по подсказке → ревью черновика → публикация (как обычно)
  → при следующем seed_corpus (или вручную) PendingAct удаляется как разрешённый
```

Никакого фонового скрейпинга и планировщика: список в admin — пассивный, всегда актуальный.

## 4. Обработка ошибок и краевые случаи

- **Ключ совпадения** для `is_resolved` — `(official_number, doc_type)`. Защищает от
  ложного «разрешения» по случайному совпадению номера у акта другого типа.
- **Дубликат в SEED_ACTS и PENDING_ACTS одновременно**: порядок в `seed_corpus` —
  сначала `SEED_ACTS` (акт появляется в корпусе), затем синхронизация pending, на
  которой `is_resolved` уже `True` → запись удаляется. Конфликта нет.
- **Нет совпадений** (акт ещё не в корпусе) — `is_resolved=False`, пункт виден. Это
  нормальное «ждём», не ошибка.
- Модель не трогает приём/публикацию — отказать ей нечем; риск только в корректности
  запроса `is_resolved`, покрывается тестами.

## 5. Тестирование

Прогон всего репозитория (`ruff check .` + `pytest` без путей, Postgres — см. lint-scope).

- `PendingAct.is_resolved`: `False` без акта; `False` при наличии только черновика;
  `True` при опубликованной текущей редакции совпадающего `(official_number, doc_type)`;
  `False` при совпадении номера, но другого `doc_type`.
- `seed_corpus`: идемпотентный upsert `PENDING_ACTS`; авто-удаление разрешённой записи
  (создать акт+публикацию → seed_corpus → `PendingAct` исчез); повторный запуск без
  изменений ничего не ломает.
- Admin: smoke — модель зарегистрирована, changelist отдаёт 200, `ingest_hint`
  рендерится с подставленным slug.

## 6. Сознательно НЕ делаем (YAGNI, с причинами)

- **Скрейпинг/headless-браузер поиска ИПС** — поиск ИПС это JS-приложение без
  серверного HTML-результата; авто-поиск хрупок и высоко-сопровождаем (против §2.1).
- **OCR скан-PDF с `publication.pravo.gov.ru`** — даёт исходную (не консолидированную)
  редакцию + риск ошибок OCR в юр. тексте; отдельная большая тема, в v1 отложена (§3).
- **Авто-создание черновика по находке** — требует надёжного авто-поиска, которого нет.
- **Email/планировщик-нудж** — постоянно видимого списка в admin достаточно; SMTP и
  фоновая задача — лишние движущиеся части. Можно добавить позже, если потребуется.

## 7. Выкатка

- Миграция применяется обычным `migrate`.
- `seed_corpus` заводит первую запись (565-ФЗ) автоматически.
- Откат — снести миграцию; данные не критичны (реестр-напоминание, не корпус).
