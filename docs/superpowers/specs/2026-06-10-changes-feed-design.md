# Лента изменений (changes feed) — Design

**Дата:** 2026-06-10. **Статус:** утверждено (v1 пункта §17 «уведомления об изменениях»).
**Контекст:** читателю нужна одна страница, где видно, что недавно опубликовано в корпусе.

## Что делаем

- `Redaction.published_at` (`DateTimeField`, null) — момент публикации; `publish()` ставит
  `timezone.now()`. Schema-миграция + data-миграция: для уже опубликованных строк backfill
  из `redaction_date` (aware datetime на полночь — лучшее доступное приближение).
- `GET /changes/`, name=`changes_feed`, view в `documents` app, `@login_required` (§10).
  Только `review_status=published`, `select_related("document")`, сортировка
  `-published_at` (nulls last), `Paginator` по 20.
- Запись ленты: название акта (ссылка на `document_detail`), «действует с»
  (`redaction_date`), «опубликовано» (`published_at`), и — если у того же документа есть
  более СТАРАЯ опубликованная редакция — ссылка «что изменилось» на `redaction_diff`
  (`/doc/<slug>/diff/<from_pk>/`, from_pk = pk предыдущей опубликованной редакции).
  Предыдущие pk считаются одним доп. запросом (словарь doc_id → отсортированные редакции),
  без N+1.
- Шаблон extends `base.html` (Pico), nav-ссылка «Изменения», пустое состояние
  «Изменений пока нет.» Пагинация — обычные ссылки, без htmx.

## Сознательно НЕ в v1 (YAGNI)

- Email/RSS/push-уведомления, подписки на конкретные акты — отложено.
- Фильтры ленты (по типу акта, дате) — отложено.
- htmx-подгрузка страниц ленты — обычные `?page=` ссылки достаточно.
