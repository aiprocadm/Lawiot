# Дизайн: автономное обнаружение подзаконных актов (полуавтомат)

**Дата:** 2026-06-17
**Статус:** утверждён пользователем, готов к плану
**Ветка:** `feature/lawiot-autonomous-discovery`

## Контекст и цель

Стратегия — «лучший справочник по трудовому праву» → **полнота корпуса** →
подзаконные акты (постановления, приказы Минтруда). Парсер таких актов уже
построен (`parse_points`, PR #41). Осталась **сложная часть — источники**: как
системе самой находить и подтягивать новые акты, не полагаясь на ручной ввод
ссылок.

### Решающая находка спайка (durable)

Два источника, каждый даёт ровно половину:

- **`publication.pravo.gov.ru`** — портал официального опубликования, имеет
  **JSON API**: `GET /api/Documents?...` (список/фильтр), `GET /api/SignatoryAuthorities`
  (органы; федеральный Минтруд = GUID `2c4929b0-9323-4541-9705-76185b9e284b`),
  `GET /file/pdf?eoNumber=<EO>` (PDF). Но **PDF — картинки-сканы** (0 шрифтов,
  340 изображений на пример-акте) → чистый текст только OCR. **Обнаружение
  автоматизируется, чистого текста нет.**
- **ИПС `pravo.gov.ru/proxy/ips/?doc_itself=&nd=<ID>`** — даёт **чистый
  консолидированный HTML** (windows-1251), которым уже питается ингест (ТК РФ).
  Но **поиск/обнаружение `nd=` не автоматизируется**: stateful JS-фреймсет,
  headless отдаёт 500 либо кэш региональных актов; связки ID между порталом и
  ИПС нет.

### Выбор пользователя

- **Точность текста важнее полной автономности** (полуавтомат): чистый HTML из
  ИПС, **без OCR** (OCR-ошибка в праве опасна).
- **Бридж к тексту:** «система пробует сама + куратор подтверждает» — резолвер
  `nd` работает **best-effort** (часто не найдёт — это нормально), куратор
  подтверждает кандидата или вставляет `nd` вручную.

## Архитектура — конвейер из 4 звеньев

```
1. ОБНАРУЖЕНИЕ            2. РЕЕСТР           3. БРИДЖ к тексту        4. ИНГЕСТ (есть)
publication API     →    PendingAct     →    nd-резолвер +       →    ingest_target +
(Минтруд, по дате)       (метаданные)        куратор подтвердил        parse_points
   автономно               автономно         best-effort + человек     автономно
```

Звено 4 уже работает (PR #41 — `parse_points` для decree/order). **Эта спека =
звенья 1–3.** Каждый компонент — одно назначение, изолированный интерфейс,
тестируется отдельно.

## Звено 1 — клиент API портала (`ingestion/publication.py`)

Изолированный сетевой слой (по образцу `ingestion/fetching.py`: httpx, в тестах —
`httpx.MockTransport`, живая сеть не нужна).

- `iter_documents(signatory_authority_id, *, since_date=None) -> Iterator[PublicationDoc]`
  — перебирает `/api/Documents` постранично (`PageSize` ∈ {10,20,50}, `Index`/номер
  страницы; точные имена параметров — захватить живой ответ на этапе плана),
  отдаёт нормализованные записи.
- `@dataclass PublicationDoc`: `eo_number`, `number` («782н»), `name`,
  `complex_name`, `document_date: date`, `signatory_authority_id`,
  `document_type_id`, `pages_count`, `reg_number`, `pdf_url`.
- Константа `FEDERAL_MINTRUD_ID = "2c4929b0-9323-4541-9705-76185b9e284b"`.
- Маппинг `document_type_id` (GUID) → `Document.DocType` (приказ/постановление):
  словарь GUID→тип; неизвестный тип → `OTHER`. GUID типов уточнить из
  `/api/DocumentTypes` или живого ответа на этапе плана.

Сеть изолирована здесь; разбор JSON чист и тестируем на сохранённой фикстуре.

## Звено 2 — обнаружение (`ingestion/discovery.py`)

- `discover(authority_ids, *, since_date=None) -> DiscoverySummary` — для каждого
  органа листает `iter_documents`, **upsert** `PendingAct` по ключу `eo_number`
  (идемпотентно — повторный обход не плодит дубли); проставляет метаданные и
  `doc_type`; **пропускает уже заведённые** акты (`PendingAct.is_resolved` —
  существует `Document` с тем же номером/типом и опубликованной редакцией).
- `DiscoverySummary` (как `SweepSummary`): сколько проверено/создано/пропущено/ошибок.
- Изоляция по органу: падение одного не валит остальные (try/except на орган,
  ошибка в summary).
- `run_discovery()` — задача django-q; идемпотентное расписание `discovery`
  (новый `DISCOVERY_CRON`, по умолчанию отдельный час, напр. `0 4 * * *`), команда
  `ensure_discovery_schedule` (по образцу `ensure_sweep_schedule`) и ручная
  команда `discover_acts`.

## Звено 3 — расширение `PendingAct` + бридж

### Модель (миграция — следующий свободный номер)

`PendingAct` уже есть (slug/title/official_number/doc_type/note/ips_search_url/
added_at + `is_resolved`). Добавляем поля; номер миграции — **следующий свободный
на момент реализации**: от `main` это `0014`, но если PR #41 (`0014_alter_article_kind`)
смержён раньше — будет `0015`. Назначить при `makemigrations`.

Поля:

- `eo_number = CharField(max_length=40, blank=True)` — EO с портала (пусто у
  вручную-добавленных). **Partial unique constraint** `UniqueConstraint(fields=["eo_number"],
  condition=~Q(eo_number=""), name="uniq_pendingact_eo")` — авто-находки не дублируются,
  ручные записи без EO не конфликтуют.
- `publication_url = URLField(blank=True)` — ссылка на акт/PDF на портале.
- `document_date = DateField(null=True, blank=True)`.
- `issuing_body = CharField(max_length=255, blank=True)`.
- `source = CharField(choices=[("auto","Авто"),("manual","Вручную")], default="manual")`.
- `ips_nd = CharField(max_length=40, blank=True)` — привязанный `nd` (резолвер/куратор).
- `resolution_status = CharField(choices=[("new","Новый"),("candidate","Есть кандидат"),
  ("bound","Привязан"),("dismissed","Отклонён")], default="new")`.

### Резолвер (`ingestion/ips_resolve.py`)

- `resolve_nd(act: PendingAct) -> ResolveResult` — **best-effort**: пытается найти
  `nd` в ИПС по номеру/названию; `@dataclass ResolveResult(candidates: list[str], note: str)`.
  Может вернуть пустой список — это штатно. Изолирован за интерфейсом, чтобы
  улучшать без переделки звеньев 1–2. Сетевые/парсинг-сбои → пустой результат.
- Первичная реализация — тонкая попытка (best-effort); **надёжный автopoиск ИПС
  не в scope** (см. YAGNI).

### Действие куратора (admin)

- Админ-список `PendingAct` показывает находки, метаданные, авто-кандидата `nd`
  (если есть) и поле для ручного ввода `ips_nd` (или ИПС-URL, из которого извлечь `nd`).
- Действие **«Привязать к ИПС и включить авто-ингест»**: из `ips_nd` строит
  ИПС-адрес `...?doc_itself=&nd=<nd>&print=1`, создаёт/обновляет `Document`
  (slug, title, official_number, doc_type, issuing_body, `source_url`=ИПС-адрес,
  `auto_ingest=True`, `auto_publish=False`), ставит `resolution_status="bound"`.
  Дальше работает звено 4 (`ingest_target` на следующем sweep или по ручной
  команде `ingest_url`).
- **Инварианты доверия:** `Document` не создаётся без действия куратора; авто-публикация
  по угаданному `nd` исключена (`auto_publish=False`).

## Обработка ошибок и изоляция

- Сбой API портала — поймать/залогировать, не валить процесс.
- Обнабор изолирован по органу.
- Резолвер пусто/ошибка → «кандидата нет», куратор вводит вручную.
- Дедуп по `eo_number` → идемпотентность.
- Никаких авто-`Document` без куратора; никакой авто-публикации по угаданному `nd`.

## Тестирование (по образцу существующих)

- **Клиент API** — `httpx.MockTransport` + сохранённая JSON-фикстура портала
  (`/api/Documents` и `/api/SignatoryAuthorities`): нормализация полей, пагинация,
  маппинг типа.
- **Обнаружение** — `discover` создаёт `PendingAct` из метаданных; дедуп по EO
  (повторный прогон не плодит); маппинг приказ/постановление; пропуск
  `is_resolved`; изоляция ошибки органа.
- **Резолвер** — извлечение кандидата из мок-ответа ИПС + мягкий пустой результат.
- **Действие админки** — создаёт `Document` с верными полями (`source_url`,
  `auto_ingest=True`, `auto_publish=False`), `resolution_status="bound"`, без публикации.

## Сознательно вне scope (YAGNI) v1

- **OCR** картинок-PDF (выбрана точность).
- **Надёжный автopoиск ИПС** — резолвер остаётся best-effort; «допиливание» поиска
  ИПС — отдельная будущая задача.
- **Другие органы** кроме федерального Минтруда — модель готова (список органов из
  `/api/SignatoryAuthorities`), стартуем с Минтруда; расширение — правкой конфига.
- **Авто-привязка без куратора** и авто-публикация по угаданному `nd` — нет.
- **Звено 4** (`ingest_target`/`parse_points`) — уже построено (PR #41).

## Зависимости

- **PR #41** (`parse_points`) — для сквозного разбора приказов/постановлений на
  выходе конвейера. Звенья 1–3 от его кода не зависят (создают `Document` с
  `auto_ingest`); сквозной эффект — после мержа #41.
- Точные имена параметров `/api/Documents` и GUID типов «приказ/постановление» —
  захватить живым ответом на этапе плана (API проверен рабочим в спайке).

## Критерии готовности

- `discover` по федеральному Минтруду создаёт `PendingAct` с метаданными,
  идемпотентно (повторный прогон — 0 новых), маппинг типов верный.
- Куратор привязывает `nd` (авто-кандидат или вручную) → создаётся
  `Document(auto_ingest=True, auto_publish=False, source_url=ИПС)`; ингест и разбор
  идут существующим звеном 4.
- Резолвер при неудаче не валит процесс (пустой кандидат).
- Весь набор тестов зелёный, `ruff check` чист.
