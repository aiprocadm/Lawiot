# Roadmap: путь к полноценной НСС-альтернативе (Гарант / Консультант / Контур.Законодательство)

Создан: 2026-06-23. **Актуализирован: 2026-06-30** (аудит реального состояния кода 4 агентами —
см. §0). Живой документ — обновлять по мере закрытия треков.

## 0. Аудит 2026-06-30 — что изменилось с 2026-06-23

За неделю проект ушёл далеко вперёд. Версия 2026-06-23 уже устарела: почти вся «очередь
разблокированных инкрементов» (§6 старой ревизии) и большинство GAP'ов матрицы — **построены и
покрыты тестами**. Аудит (read-only, 4 параллельных агента по подсистемам: assistant / search /
reader-nav / corpus-ingestion) установил фактическое состояние. Ключевые сдвиги:

- **Корпус: ~93 акта** (было «единицы»): ~10 законов трудового права (ТК РФ, 426-ФЗ, 10-ФЗ, 82-ФЗ,
  125-ФЗ, 4520-1, 165-ФЗ, 81-ФЗ, 92-ФЗ, 156-ФЗ) + 23 кодекса РФ + 58 приказов Минтруда (архив,
  status=repealed) + ПП № 2464; ещё ~14 в `PendingAct`.
- **Судебная практика — РЕАЛИЗОВАНА** как ручная курация: приложение `practice/`, модель
  `CourtDecision` (FK на Document + `article_number`), вывод на странице статьи. Авто-парсера нет
  и не планируется (правовая интерпретация — человек).
- **Личный слой — РЕАЛИЗОВАН**: `bookmarks/` (избранное), `notes/` (заметки, в т.ч. к статье),
  `history/` (ViewHistory с троттлингом 1×/мин). Нет только единой страницы-кабинета.
- **Экспорт — РЕАЛИЗОВАН**: DOCX (`document_export_docx`) + версия для печати (`document_print`).
- **Межактовые ссылки — РЕАЛИЗОВАНЫ**: `documents/refs.py` резолвит и «N-ФЗ», и имена кодексов
  (со словоформами) в URL актов корпуса; внутри-актовые «ст. N» тоже.
- **AI: все «срезы» закрыты** — стриминг (SSE), многоходовой диалог (сессия, до 10 ходов),
  семантический поиск (pgvector + e5-small, гибрид с FTS), AI-объяснение diff, разъяснение статьи
  «простыми словами», гард-рейлы (cap 12k, анти-инъекция, проверка цитат, лог токенов, таймаут).
- **Поиск**: FTS russian + лемматизация (pymorphy3) + синонимы + поиск-внутри-документа +
  семантика + сортировка + фильтры (тип/статус/орган/даты) + пагинация.

Вывод: **разблокированная автономная очередь фактически исчерпана.** Оставшиеся крупные приросты
заблокированы внешними решениями (источники корпуса, легальный источник судебки, юристы-редакторы).
Что ещё можно строить без действий user'а — небольшие UX/операционные доработки (§6).

## 1. Стратегия (без изменений)

НЕ догонять классические СПС по **объёму** (миллионы документов, штат редакторов — нереально
для нашего масштаба), а быть **лучшим в нише трудового права** через **глубину + AI**.
Дифференциатор — AI-ассистент с заземлением в корпус (классические СПС только подступаются к
этому). Полнота корпуса — догоняем точечно по трудовому праву, не «всё подряд».

«Полноценная альтернатива» в нашей трактовке = в нише трудового права пользователь получает:
актуальный текст актов + поиск + связи + изменения + **обоснованный AI-ответ** не хуже, чем в
большой СПС, плюс уникальную глубину (внутренние ссылки, diff редакций, консолидация).

## 3. Треки

### A. Объём корпуса трудового права — НАИБОЛЬШИЙ прирост, но ЗАБЛОКИРОВАН источником
- **Инфраструктура полностью готова и обкатана на реальных данных:** `seed_corpus`, `ingest_url`,
  `sweep_targets` (django-q2), авто-консолидация с гейтом публикации (`_is_safe_to_publish`:
  блок при 0 статей или <80% от текущего числа), change-detection по нормализованному хэшу,
  semi-auto discovery (`discover_acts` → `PendingAct` → `bind_to_ips`). Дать ID — конвейер наполняет.
- **Блокер: источник.** ИПС pravo.gov.ru (`doc_itself&nd=<ID>&print=1`) даёт чистый HTML, но
  поиск `nd=` headless нестабилен (stateful JS-фреймсет → 500; `ips_resolve.resolve_nd` штатно
  возвращает пустой результат, куратор вводит вручную). `publication.pravo.gov.ru` имеет JSON-API
  (discovery работает), но многие документы там — PDF-сканы без текста.
- **Конкретные заблокированные кандидаты:** 255-ФЗ (пособия по нетрудоспособности — ключевой,
  headless nd= не резолвится), 8 приказов по охране труда (только PDF-сканы → нужен OCR или nd=).
- **Что нужно от user'а:** `nd=`-URL целевых актов из реального браузера, ЛИБО Claude-in-Chrome
  (тогда обнаружение/резолв автоматизируется). Без этого трек стоит.

### B. Судебная практика — модель есть, нужен КОНТЕНТ и решение по источнику
- **Реализовано:** `practice/CourtDecision` (суд, дата, № дела, позиция, source_url, FK на Document +
  `article_number`), вывод на странице статьи, курируется через админку. Авто-парсера нет (намеренно).
- **Что осталось:** наполнить ключевыми Постановлениями Пленума ВС РФ по трудовым спорам (ручной
  набор). При желании авто-сбора — нужен легальный источник (sudact/ГАС «Правосудие» — антибот/без
  API; КонсультантПлюс/Гарант запрещены §1). Решение user'а: курировать вручную vs искать источник.

### C. AI-глубина — ВСЕ ЗАПЛАНИРОВАННЫЕ СРЕЗЫ ЗАКРЫТЫ
- ✅ Срез 2 стриминг (SSE), ✅ срез 3 диалог, ✅ срез 4 семантика, ✅ срез 5 AI-diff,
  ✅ бонус: разъяснение статьи «простыми словами», ✅ гард-рейлы.
- Возможные доработки (не из исходного плана, см. §6): rate-limiting, БД-персист диалога между
  сессиями, проверка цитат через structured output вместо regex.

### D. Reader/навигация — ПОЛНОСТЬЮ РАЗБЛОКИРОВАНО И ПОСТРОЕНО
- ✅ Печать (`@media print` + `/doc/<slug>/print/`), ✅ DOCX-экспорт, ✅ межактовые ссылки,
  ✅ закладки/заметки/история. Остаток — единая страница-кабинет (мелкий UX, §6).
- **Редакция на дату** — инфраструктура (несколько `Redaction`) есть; нужна 2-я опубликованная
  редакция реального акта (демон поймает реальную поправку со временем).

### E. Редакторский слой (нужны люди)
- Комментарии/аннотации к статьям, обзоры — контент юристов-редакторов. Вне автономного scope;
  продуктовое решение user'а (нанимать/писать самому).

### F. Надёжность/мониторинг (в осн. готово)
- Авто-консолидация + B1 (нормализованный хэш) на main; sweep/discovery через django-q2
  (`ensure_sweep_schedule` / `ensure_discovery_schedule`). Демон qcluster `docker compose up -d`
  для боевого daily-sweep — инфра-хвост. Команда `corpus_health` для проверки целостности.
